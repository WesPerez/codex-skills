# 源码审计与主方案

仅在用户明确要求为一个长期项目建立 canonical plan、或现有长期项目需要重新基线化时读取。一次性代码审计、普通详细方案和单次实现不因“全面”或“详细”自动进入本流程。

## 1. 审计入口

开始时先获得这些事实，能发现的不要询问用户：

```text
仓库根目录
适用 AGENTS.md
当前分支、HEAD、upstream、ahead/behind、dirty/ignored、stash
最近提交和历史迁移
语言、框架、构建系统、包管理器
主入口、数据目录、持久化位置、配置和凭据边界
现有测试、审计、CI、构建、发布命令
当前进程、服务、端口、窗口和外部系统
用户指定的历史线程、日志、截图和报告
```

执行顺序优先只读：`rg --files`、`rg`、Git 查询、文件读取、测试列表、进程查询。不要先修改或清理。

不要把“全面”理解成先研究数小时才第一次反馈。先交付足以改变当前决策的临时事实矩阵、P0 风险、证据缺口和 provisional plan；随后边验证纵向切片边补深度。对不影响当前决策的问题放入 backlog。

## 2. 需求提炼

把原始需求重写成下面的需求树：

```text
根本目标
├─ 核心用户工作流
├─ 必须具备的能力
├─ 质量属性
│  ├─ 安全
│  ├─ 性能
│  ├─ 可恢复
│  ├─ 可维护
│  └─ 可观察
├─ 禁止事项
├─ 非目标
├─ 实际环境约束
└─ 可验证完成定义
```

对每项需求标记：`must`、`should`、`could`、`out_of_scope`。发现冲突时说明权衡，不凭偏好静默取舍。

特别识别用户真正担心的事情：

- 看不到进度。
- 代码存在但从未运行。
- 自动化抢占真实鼠标键盘。
- 数据迁移破坏真实配置。
- 中断后重复发送输入、重复推送或重复写入。
- UI 功能很多但路径复杂、默认窗口不可达。
- 样例数量代替真实可用性。

## 3. 当前事实矩阵

不要给单一总完成百分比。最少建立：

| 维度 | 状态 | 证据 | 缺口 |
|---|---|---|---|
| 需求/设计 | not_started/partial/passed | 文档、决策 | 未决选择 |
| 代码表面能力 | ... | 模块、入口、测试 | 断链/占位 |
| 自动测试 | ... | 命令、退出码 | 未覆盖风险 |
| 当前构建 | ... | artifact hash | 未打包平台 |
| 当前应用启动 | ... | PID、exe hash、截图 | 旧进程/未启动 |
| 真实业务动作 | ... | 外部后置状态 | 只有 preflight |
| 失败恢复 | ... | 故障注入报告 | 未测超时/中断 |
| 持久化/重启 | ... | 重启回查 | 迁移未验证 |
| 多实例/并发 | ... | 隔离报告 | 共享状态冲突 |

`partial` 必须说明已证明和未证明的边界。

## 4. 源码与调用链审计

按真实工作流从外向内追踪：

```text
用户操作
-> UI 事件
-> 状态管理/命令构造
-> IPC/API/任务模型
-> runner/服务/数据库/平台层
-> 外部系统动作
-> 后置状态
-> 运行记录/失败报告/持久化
```

对每个关键能力检查：

- 数据模型是否能表达需求。
- UI 是否真正能创建和编辑该模型。
- 持久化是否保存、迁移和恢复该字段。
- runner 是否执行该字段。
- 失败路径是否可定位和恢复。
- 测试是否覆盖成功、失败、兼容和边界。
- 文档和样例是否与实现一致。

记录关键模块地图：

| 模块 | 责任 | 上游 | 下游 | 风险 | 测试 |
|---|---|---|---|---|---|

大型单文件不是自动重构理由。只有耦合阻碍当前目标、测试或安全时才拆分。

## 5. 多余代码判断

删除前至少核对：

1. `rg` 调用和导入。
2. 动态注册、字符串反射、配置引用和构建入口。
3. 测试、fixture、文档、样例和迁移路径。
4. 平台条件编译和 feature flag。
5. Git 历史说明它是否用于兼容旧数据。
6. 删除后构建、测试、运行和持久化影响。

分类：

```text
confirmed_dead      明确无调用且无兼容价值
duplicate           与现有实现重复
obsolete            已被新实现取代
broken_link         UI/模型/runner 某段断链
legacy_required     仅用于迁移或兼容，必须保留
uncertain           证据不足，暂不删除
```

不要因为名字旧、目录像临时、代码难看或没有当前测试就删除。

## 6. 参考项目和外部资料

阅读参考项目时只抽取：

- 数据模型和 schema 演进。
- 任务/步骤/重试/条件/循环的表达方式。
- 素材、目标、ROI、OCR 和预览工作流。
- 队列、并发和中断恢复。
- 用户操作路径和失败可见性。

不要盲目复制：许可、运行环境、权限模型、技术栈、输入方式和用户群可能不同。

每项外部结论使用证据表：

| 结论 | 来源性质 | 链接/版本 | 当前项目适用性 | 本项目推导 |
|---|---|---|---|---|

社区帖子只能说明“有人报告该故障模式”，不能升级为官方承诺或普遍事实。

研究停止条件：关键决策已有一条官方/一手能力边界，加一条成熟实现或当前项目实测互证；继续搜索只增加相似案例而不改变方案；剩余不确定性已写入风险和验证计划。达到条件后开始最小纵向切片，不追求“读完整个互联网”。

## 7. 方案决策

每个关键选择记录：

```text
问题
上下文和约束
候选 A/B/C
各自收益、成本、风险和迁移影响
选择
为什么不选其它方案
验证方式
可逆性和回退点
```

需要 ADR 的情况：跨模块架构、安全边界、持久化方案、不可逆迁移、外部输入方式、并发模型。不要为小样式调整写 ADR。

## 8. 主方案模板

```markdown
# 项目审计与实施总方案

## 1. 文档目的和权威边界
## 2. 根本需求、非目标和用户工作流
## 3. 当前 Git、运行环境和历史事实
## 4. 当前完成度分轴
## 5. 源码架构、调用链和数据流
## 6. 功能现状与断链
## 7. 多余/过时/重复代码候选
## 8. 安全、权限、持久化和迁移风险
## 9. UI/UX 审计
## 10. 外部资料、参考项目和结论边界
## 11. 目标架构和数据模型
## 12. 用户工作台和关键交互方案
## 13. 执行、队列、并发和失败恢复方案
## 14. P0-Pn 阶段路线
## 15. 每阶段纵向切片和建议 commit
## 16. 测试、构建、实机和失败矩阵
## 17. 进度台账与中断恢复
## 18. 风险、已知限制和待用户决策
## 19. 最终交付定义
```

文档开头必须写权威顺序和生成日期。事实与建议分开书写。

## 9. 阶段与纵向切片

阶段按风险依赖排序，不按页面或目录机械分组：

```text
P0 数据/环境保护和可重复基线
P1 安全与身份硬化
P2 最小用户工作流和可达 UI
P3 核心引擎/捕获/识别
P4 第一个真实端到端纵向任务
P5 持久化和迁移
P6 更多真实任务
P7 多实例/并发/队列
P8 失败矩阵和发布门
P9 清理、提交、推送和发布
```

阶段名称应按项目调整，但顺序要先保护数据和安全，再扩功能。

每个切片必须有：范围、非目标、安全边界、可见结果、验收条件、证据类别、预计范围、失败回退和建议 commit。

## 10. 常见失败

- 一开始就重构，尚未证明真实问题。
- 用代码搜索结果代替运行验证。
- 用样例数量代替实际业务闭环。
- 在旧应用进程上观察 UI，却声称当前源码已启动。
- 只写“大而全”方案，没有唯一下一动作。
- 方案按文件罗列，没有用户纵向工作流。
- 精确 ETA 没有假设和置信度。
- 为了显示进度制造大量无意义事件。
- 方案和 STATUS 混成一个文件，导致恢复者找不到当前停点。
