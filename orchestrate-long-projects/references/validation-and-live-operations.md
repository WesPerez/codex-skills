## 1. 按失败模式选择最低充分证据

L0-L10 是证据工具箱，不是每个任务都必须顺序跑完的固定阶梯。先列出本次变更可能出现的失败模式，再为每个失败模式选择能直接证伪它的最低成本证据；只有当前层无法覆盖风险、结果冲突或变更触及真实边界时才升级。

```text
L0 源码/配置审计
L1 纯函数和单元测试
L2 模块/集成测试
L3 静态检查、lint、clippy/typecheck
L4 当前源码生产构建
L5 当前二进制/服务启动
L6 UI/API 最小 smoke
L7 最小真实外部动作
L8 业务后置状态
L9 失败、并发、重启和恢复
L10 发布/安装包/目标环境
```

例如纯文档措辞通常止于 L0；纯函数修复可止于 L1；构建配置变更需要 L3-L4；启动路径需要 L5-L6；外部系统、持久化、并发、恢复或发布风险才选择 L7-L10。每项证据都说明它覆盖的失败模式和未覆盖风险；L4 通过不等于 L5-L10 通过，也不意味着必须默认执行 L5-L10。

## 2. 证据标准

每条强证据绑定：

- 完整 commit/HEAD。
- dirty entries 和工作树 fingerprint。
- 命令参数、cwd、环境身份和退出码。
- 开始/结束时间。
- artifact path、hash、大小和保留策略。
- 运行二进制 hash、PID/启动时间或服务版本。
- 外部目标身份。
- 前置与后置状态。

截图需显示可判断业务结果的真实内容，不用模糊、裁切过度或纯装饰图。

## 3. UI 和视觉检查

UI 改动必须实际启动当前版本。至少检查：

- 默认窗口、最小窗口、常见桌面分辨率。
- 关键流程是否在首屏或少量步骤可达。
- 文本溢出、遮挡、滚动陷阱、卡片嵌套和动态布局跳动。
- 空、加载、失败、权限不足、缺数据、部分就绪和完成状态。
- 用户常用动作的点击数和回退路径。
- UI 显示的 readiness 是否与 runner 真实门禁一致。

浏览器/Web UI 使用 Playwright 或浏览器工具截图和交互；桌面 UI 使用 Computer Use/平台工具。归属不明页签和窗口不关闭。

## 4. 外部输入和桌面自动化

默认不执行真实输入。执行前必须：

- 用户授权在当前任务范围内成立。
- 目标窗口/设备身份包括句柄、PID、进程、标题、尺寸、权限和启动实例。
- 当前二进制和配置 hash 已验证。
- 有 pre-live checkpoint、后置验证方案和停止方式。
- 前台焦点、鼠标位置、用户并行操作保护已设计。

消息/API 返回成功通常只证明请求被接收，不证明业务完成。使用截图、OCR、状态查询或人工观察确认后置状态。

后台定向输入不等于符合目标应用条款、反作弊或业务许可；单独披露合规风险。

## 5. 数据库与持久化

未确认数据库身份时按生产库处理；生产库只读。

开发/测试写入前确认：环境身份、备份、样本、影响范围、事务、回查和回滚。批量修改或迁移即使在测试库也先说明影响。

文件/AppData 持久化检查：

- schema/version。
- 原子写入和 fsync/replace。
- 主文件、备份、mtime、size 和 hash。
- future schema 的只读保护。
- 迁移幂等和失败恢复。
- 重启后任务、素材、队列、设置和历史保留。

真实数据 fixture 必须匿名化后才能提交。

## 6. 多实例和并发

至少区分：同一资源内串行、不同资源间并行、全局共享状态。

使用稳定并发键：窗口/设备 ID、数据库租户、工作区 ID、仓库写入 lease。仅 PID 或标题不够。

验证：

- A/B 实例不同任务同时执行。
- 同一实例不会并发输入。
- 暂停/继续只影响目标实例。
- 一个实例失败不污染另一个。
- 队列顺序和重试不会重复已成功步骤。

## 7. 失败矩阵

按功能选择真实风险：

```text
依赖/构建失败
权限不足
目标消失或身份变化
超时
图像/OCR 不匹配
数据库/文件不可写
schema 不兼容
网络中断
进程退出
动作执行后回执丢失
队列暂停/取消/恢复
应用重启
第二实例并发
```

每个失败验证：用户可见错误、结构化报告、重试策略、恢复点、不会误操作其他目标。

## 8. 清理和进程安全

清理前回答：

1. 具体目标是什么。
2. 是否由本轮明确创建/启动。
3. 直接证据是什么。
4. 是否可能属于用户、IDE、浏览器、其他 Agent 或旧任务。
5. 误清理的影响。

不能证明属于本轮就保留。禁止按关键词批量杀进程、清空所有缓存、模糊删除目录或关闭归属不明页签。

测试超时要考虑子进程树。只终止直接子进程可能遗留 Node、Cargo、浏览器或服务；优先使用 Job Object/进程组并登记 PID、命令、启动时间和工作目录。

## 9. 发布门

发布前至少：

- clean 或明确可解释的工作树。
- 全部 required gate 当前有效。
- release 构建和安装包验证。
- 当前版本启动截图/日志。
- 数据迁移和回退演练。
- 外部系统最小真实闭环。
- 失败、并发、重启矩阵。
- 文档、版本、commit、tag 和远端 ref 一致。
- 生成物、运行进程、配置变化和保留项报告。

不要因为最后一个测试通过就跳过中间 gate 聚合。
