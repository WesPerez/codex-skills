# 风险驱动的证据选择

## 核心原则

验证目标不是达到最高层，而是用最快、可靠、可定位的证据降低具体风险。高层证据更接近真实使用，但通常更慢、更不稳定、根因更难隔离；只有低层无法证明关键行为时才增加高层证据。

## 冲突处理规则

1. 先确认来源是否讨论同一对象、同一版本和同一结论；不同层面的建议不视为真正冲突。
2. 产品结构、触发和工具契约优先使用 A1；通用测试术语优先 A2；工程取舍由 A3-A4 补充；社区帖子只用于发现候选失败模式，不进入规范矩阵，除非被官方资料或可复现实证闭环。
3. 当前可复现的产品行为与 A1 文档冲突时，只对已记录的版本、环境和最小复现暂时优先，并标记“文档/实现冲突”，不得外推到其他机器或版本。
4. 安全、权限、生产写入和不可逆外部状态始终服从更严格的适用规则；不能用低权威经验降低安全门。
5. E0-E4 是本地综合：E0 对应结构静态证据，E1 对应确定性本地行为，E2 对应边界契约，E3 对应最小真实集成，E4 对应独立解释/交接。任何来源都没有单独发布这套五级命名。

## 纳入新来源的门槛

新增来源前必须回答：发布者是谁、原文支持哪一句结论、适用于什么对象和版本、最后核验时间、与现有来源是否独立、什么变化会使它失效。只重复已有观点、只有搜索摘要、无法定位原文、匿名转载或缺少可复现细节的材料不进入治理矩阵；可保留为待核验线索。

## 选择示例

### 只修改 description

风险：误触发或漏触发。

证据：E0 frontmatter 校验，加 2-4 个由主代理完成的应触发/不应触发请求推演。该矩阵本身就是代表性通过证据，通常不需要网络、真实系统或子代理。

### 新增 PowerShell 脚本

风险：编码、参数绑定、退出码、路径和输出格式。

证据：E0 + E1。使用固定输入跑成功案例和坏输入案例。只有脚本核心依赖真实网络响应时，再增加一个 E3 只读 smoke。

### 浏览器登录流程技能

风险：DOM、登录态、标签页归属和真实交互顺序。

证据：E0 + E2 页面/工具契约 + 一个 E3 最小登录后只读流程。删除、提交、发送等外部副作用停在安全门前。

### 数据库迁移技能

风险：环境误判、结构或数据破坏、锁和回滚失败。

证据：E0 + E1 SQL/脚本静态和 fixture + E2 schema/迁移契约。只有已确认隔离、可回滚的本地或测试库才允许 E3。生产写入永不作为技能验证。

### 跨技能编排

风险：交接丢失产物、状态或停止条件。

证据：先用 E2 交接样例验证输入输出。只有问题是“新代理冷启动能否正确理解”时才增加 E4；不要因为存在两个技能就自动启动子代理。

## 证据失效条件

已有证据出现以下情况时应重新验证：

- 技能正文、脚本、模板或依赖版本发生相关变化。
- 解释器、操作系统、浏览器、外部 API 或文件格式版本变化。
- 真实端点、认证方式、权限或页面结构变化。
- 发现新的失败模式、事故或用户反馈。
- 原证据来自不同环境，或无法证明产物与当前版本对应。

纯措辞或无关引用修改不会自动使所有运行证据失效。
