# 技能改进器

## 目的

用与风险相称的最低充分证据证明技能的受影响能力，而不是把每个技能都拖入完整真实演练。验证始终从最快、最稳定、最容易定位失败的证据开始；只有低层证据无法覆盖核心风险时才升级。

验证不是“快速/标准/完整”三选一套餐。静态、确定性运行、契约、真实集成和独立解释是可组合证据层，每项能力可以停在不同层。

详细选择示例、来源治理矩阵和冲突规则见 `references/evidence-selection.md`。当结论依赖外部资料、行业经验或本地案例时必须读取该引用；纯结构校验不必加载。

## 工作契约

输入：目标技能路径或技能名、本次变更或审查请求、相关脚本/引用/产物、可选用户式任务，以及安全和成本边界。

输出：受影响能力、主要失败模式、选择的证据组合、实际结果、技能修补、残余风险、证据失效条件和清理/状态变化。

状态标签：

- `结构有效`：E0 通过，只证明技能结构、引用和静态边界。
- `范围内行为已验证`：受影响能力已由 E0-E2 的最低充分证据覆盖。
- `真实集成已验证`：至少一个必要的 E3 最小真实切片通过。
- `独立解释已验证`：必要的 E4 冷启动或交接验证通过。
- `部分验证-存在残余风险`：安全门、环境、授权或依赖阻止更高层证据。
- `阻塞-缺少产物`、`阻塞-缺少安全环境`、`阻塞-需要用户授权`。

若另一个技能拥有具体领域流程，点名该技能并传递目标、产物、边界和状态，不复制其完整规则。

## 完成标准

满足以下条件即可完成，不要求所有技能都跑真实任务：

1. frontmatter 的触发和排除范围清楚。
2. 正文、引用、脚本和交接契约可复用，没有一次性线程结论或陈旧路径。
3. 本次受影响的能力已列出主要失败模式。
4. 每个重要失败模式都有能够直接观察它的最低充分证据，或有明确安全阻塞。
5. 涉及分支、错误处理或安全边界时，至少包含一个成功案例和一个最重要的失败/安全案例。
6. 修补后只重跑受影响证据和 E0 基线；不默认重演整条流程。
7. 最终报告记录未验证项、原因和现有证据何时失效。

`quick_validate.py` 只能证明结构；E3/E4 也不能替代低层的精确错误定位。完成标准是证据覆盖风险，不是达到最高层。

## 工作流程

### 1. 重建上下文

完整读取目标技能的 `SKILL.md`，以及直接影响本次判断的脚本、引用、模板、`agents/openai.yaml` 和伴随技能。

记录：

- 应触发与不应触发的请求。
- 本次实际变更和受影响能力。
- 可执行运行面、外部依赖和有状态系统。
- 安全门、授权、回滚和清理边界。
- 已有证据的来源、版本、日期和适用范围。

外部依据不能只记录 URL。还要记录权威层级、它直接支持的具体结论、不能外推的边界、最后核验日期、冲突处理和失效条件。E0-E4 是结合产品约束、风险基础测试和分层测试经验形成的本地工程模型，不是 OpenAI、ISTQB、Microsoft、Google 或 Martin Fowler 发布的统一标准。

移动或重命名时额外验证目录名、frontmatter `name`、`$skill-name`、UI 元数据、引用路径和重复技能名。

### 2. 建立风险到证据表

不要先选验证等级。逐项写出：

```text
能力声明 | 主要失败模式 | 影响/不确定性 | 最低可观察证据 | 安全停止条件
```

常见能力声明：

- 技能能被正确触发，并在相似请求中不误触发。
- 指令能被未来代理无歧义执行。
- 脚本可解析、参数可绑定、编码和退出码正确。
- 模板或生成文件保持结构和视觉契约。
- 外部接口、文件格式或工具调用符合契约。
- 安全过滤、生产边界和破坏性门能够阻止危险动作。
- 技能链交接不会丢失状态、产物和停止条件。

优先考虑失败影响、发生可能性、变更不确定性、可逆性、隔离程度、依赖波动性和证据新鲜度。不要使用看似精确但无法校准的数字风险分。

来源冲突时先比较适用对象和结论粒度，再按“当前可验证的产品契约与本地行为 > 产品官方文档 > 正式标准 > 大型工程实践 > 专家经验 > 社区线索”的顺序裁决。低层来源可以补充实施经验，不能推翻更高层来源明确规定的产品、安全或工具契约；本地行为与官方文档冲突时只对当前版本和环境优先，并记录版本、复现证据和上报/复核条件。

### 3. 选择最低充分证据

按需组合以下证据阶梯：

#### E0 静态基线

所有技能修改都执行：

- frontmatter、命名和 description 触发范围。
- 引用、脚本、模板和伴随技能是否存在。
- 陈旧绝对路径、重复说明、一次性内容和不安全指令。
- `agents/openai.yaml` 与技能范围是否一致。
- `quick_validate.py` 或对应结构校验。
- 由主代理在当前线程完成的应触发/不应触发请求矩阵和步骤推演仍属于 E0；只有需要独立代理冷启动判断时才是 E4。

仅修改措辞、路由、示例或输出说明，且不改变运行行为时，E0 加针对性文本检查通常已经足够。

#### E1 本地确定性证据

技能含脚本、模板、解析器或生成逻辑时执行：

- 语法、`--help`、参数绑定、编码和退出码。
- 固定 fixture 的成功案例。
- 坏输入、缺字段、敏感值或错误路径案例。
- 输出格式、幂等性和不会误写/误删的边界。

优先使用本地 fixture，不为可在本地复现的失败调用真实外部系统。

#### E2 边界契约证据

行为依赖接口、文件格式、工具或技能交接时执行：

- 固定模拟响应、公开 schema、代表性文件、工具元数据或不依赖实时可变状态的契约调用。
- 请求参数、响应字段、错误语义和超时路径。
- 上下游交接的输入、输出、状态标签和停止条件。

契约测试能覆盖接口风险时，不立即升级完整 E2E。

只要证据依赖实时可变的外部服务、页面、认证、数据库或运行状态，即使操作只读，也归入 E3。

#### E3 最小真实切片

只有核心能力不能由 E0-E2 证明时执行一个最小切片：

- 真实网络或 API 响应是技能核心。
- 浏览器、认证、渲染、数据库只读结果或运行时版本会改变行为。
- 需要证明技能到真实工具再到结果的最短关键路径。
- 修改了高影响安全边界，低层成功/失败案例仍不足，并且存在安全的非生产或只读验证面。
- 现有真实证据过期、依赖版本变化或来源不匹配。

E3 默认只覆盖一个代表性成功路径和必要的一个失败/安全路径。不要遍历所有数据、所有页面或所有外部系统。

#### E4 独立解释与交接

仅当待证明能力本身是冷启动理解时使用子代理或独立线程：

- description 的触发/排除是否能被独立代理正确判断。
- 新代理能否仅凭技能和原始任务完成流程。
- 跨技能交接或长上下文恢复是否丢状态。
- 主代理运行会受到当前线程答案或诊断泄漏影响。

使用前必须回答：“E4 能发现哪一种 E0-E3 无法发现的具体缺陷？”没有明确答案就不使用子代理。

### 4. 应用安全门

生产写入、破坏性操作、付费、发送消息、权限变更、凭据导出和不可逆外部状态不会因为验证层升级而获得授权。

无法安全演练时，证明到安全门前，记录缺少的环境、授权、回滚或隔离条件，并标记 `部分验证-存在残余风险`。不要把安全阻塞误报为技能失败，也不要为追求“完整”越权。

### 5. 执行、修补和停止

1. 先执行 E0，再执行风险表要求的最低层。
2. 保留精确命令、输出、截图、日志或产物。
3. 发现缺陷后修补技能。
4. 重跑 E0 和受影响的证据，不重跑无关真实流程。
5. 新增验证只有在覆盖新的失败模式或使残余风险显著下降时才执行。
6. 当每个重要风险已有直接证据或安全阻塞时停止。

如果已经存在同一技能版本、同一依赖契约和同一运行环境的近期证据，且没有失效因素，可以复用并说明来源。外部服务不可用时保留残余风险，不无限重试。

## 成本控制

- 默认不启动子代理，不默认执行 E3/E4。
- 不重复读取与受影响能力无关的大型引用。
- 脚本多且同构时测试一个代表样本和公共共享组件。
- 修补只重跑最小受影响闭环。
- 完整真实流程明显比修改本身昂贵时，先说明它要发现的具体失败模式；无法说明新增信息就跳过。
- 需要第二个真实切片、第二次独立重演或多个外部系统时，必须说明第一个证据未覆盖的风险。

## 最低证据包

每次交付至少包含：

- E0 结果。
- 每个受影响能力在所选最低层的一个代表性通过证据；纯 description 修改的 E0 触发矩阵可作为该证据，不要求额外 E1-E4。
- 存在条件分支或安全边界时的一个失败/安全证据。
- 修复前失败和修复后通过的回归证据；无法保留时说明原因。
- 未验证项、原因和证据失效条件。

## 输出格式

```markdown
## 技能改进结果

- 目标技能：
- 受影响能力：
- 证据组合：E0 / E1 / E2 / E3 / E4
- 状态：结构有效 / 范围内行为已验证 / 真实集成已验证 / 独立解释已验证 / 部分验证-存在残余风险
- 关键证据：
- 技能变更：
- 跳过的高成本验证及原因：
- 剩余风险与失效条件：
- 清理和外部状态：
```

## 反模式

- 不要把固定三档当成验证套餐。
- 不要规定所有新技能必须真实演练。
- 不要因为 E3/E4 更接近真实就认为它们总是更有价值。
- 不要用完整 E2E 覆盖本地 fixture 能精确发现的错误。
- 不要重复测试同一失败模式来制造“全面”感。
- 不要把 smoke 当成完整业务验收。
- 不要为了省时忽略高影响安全边界；低成本不是低标准。
- 不要默认使用子代理；只有唯一信息增益才使用。
- 不要清理没有直接归属证据的文件、日志、进程或外部状态。
