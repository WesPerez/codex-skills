# 技能改进器

## 目的

把草稿技能或已修改技能打磨成可运行、可验证、可交接的技能。技能不是 Markdown 能解析就算好，也不是逻辑听起来合理就算好；它必须能指导 Codex 完成一个接近真实的任务，产出证据，暴露缺失说明，并经得起修补后复测。

这个技能适用于任何领域。遇到数据库、生产系统、浏览器、IDE 重启、权限、文件删除或业务流程等高风险场景时，真实演练必须遵守该领域的安全规则，只在真正需要外部状态、用户授权或安全样本时停止。

## 工作契约

输入：目标技能路径或技能名、触发本次审查的用户请求、可选示例任务、相关产物，以及用户声明的安全/范围边界。

输出：问题发现、已授权的技能修改、验证证据、真实演练结果、剩余缺口，以及清理/状态变更报告。

状态标签：`只读审计`、`已修补`、`已验证`、`部分真实演练`、`阻塞-缺少产物`、`阻塞-缺少安全运行环境`、`阻塞-需要用户授权`。

交接规则：如果另一个技能拥有具体领域流程，点名该技能，把具体产物、请求和边界传过去，并保留它的状态标签，不要把强状态压缩成模糊总结。

如果用户只是要求检查、审计或找缺口，默认从 `只读审计` 开始：检查并报告发现；只有当请求合理授权时，才修补明显、低风险的技能文档问题。若用户要求改进、修复、加固或让技能完整，则修补技能并重跑最小受影响验证闭环。

## 完成标准

只有这些条件都成立，才能认为目标技能已完成：

1. frontmatter 里的触发描述清楚，未来代理只读描述就能判断何时使用。
2. 正文只包含可复用、领域相关的流程，避免当前线程的一次性任务名、临时路径或局部结论。
3. 技能说明输入、输出、状态标签，以及与相关技能或工具的交接契约。
4. 技能已经在真实任务或最小安全真实切片上演练过。
5. 演练产生原始证据：命令输出、日志、截图、数据库只读回查、文件输出、报告，或明确的安全阻塞。
6. 演练暴露的失败、歧义、缺失交接、重复规则或不安全指令已经修补。
7. 修补后重新运行相关验证。
8. 最终报告记录测试了什么、改了什么、还有什么没测，以及为什么没测。

语法校验只能证明格式，不足以证明技能完成。

## 工作流程

### 1. 重建技能上下文

完整读取目标技能的 `SKILL.md`。如果它直接引用脚本、参考资料、模板或伴随技能，并且这些内容会影响本次评价，也要读取对应文件。

记录：

- 目标技能路径。
- 应触发该技能的用户请求类型。
- 它调用的伴随技能或工具。
- 它可能触碰的有状态系统，例如数据库、浏览器、IDE、终端、外部 API、文件系统或生成产物。

如果目标技能属于技能链，识别链路入口、下游技能、交接产物、返回状态标签和真正停止条件。

对移动或重命名过的技能，还要验证：

- 文件夹名、frontmatter `name` 和 `$skill-name` 提示一致。
- 用户可见技能存在 `agents/openai.yaml`，且展示文案与当前范围一致。
- 必需文件中没有陈旧绝对路径、旧技能名或已删除位置。
- 直接引用的脚本、参考资料、资产、模板或伴随技能仍然存在。
- 旧位置是有意不存在，或有意保留为独立副本；不能出现会混淆调用的重复技能名。

### 2. 审查复用性和边界

从多个角度检查技能：

- 触发清晰度：只看 description 就能判断何时使用。
- 范围：正文避免一次性业务标签、临时目录、当前线程产物或用户专属措辞，除非技能本来就是项目专用技能。
- 结构：核心流程在 `SKILL.md`；细节过长时拆到引用文件，并在正文说明何时读取。
- 去重：详细规则放在拥有该领域的技能里，其他技能通过交接契约调用，不复制整套流程。
- 安全：未知环境写入、生产写入、凭据暴露、破坏性清理、外部副作用和授权缺口必须有明确边界。
- 证据：领域需要时，技能必须要求源码、运行时、浏览器、数据库、文件、日志或官方文档证据。
- 上下文续航：长流程要说明何时记录恢复状态，避免上下文压缩后丢失关键证据。
- 失败处理：软阻塞要给出下一条可执行路径，只有真正边界才能停止。

如果发现明显且低风险的问题，先修补再做真实演练。

### 3. 设计最小真实演练

选择仍能触达技能核心难点的最小任务。优先使用真实项目文件和真实运行面，不用躲开关键风险的玩具示例。

演练必须包含：

- 用户式请求。
- 明确的技能入口。
- 预期下游技能或工具。
- 要收集的证据。
- 安全停止规则。
- 真实流程中不存在的层面要写明 `不适用`，例如无 SQL、无浏览器面、无运行旧系统、无代码改动。

高风险系统默认只读。不要执行生产写入。开发/测试写入也必须具备环境确认、样本、备份、回查、回滚和授权。

如果完整流程含不安全动作，证明到安全门前，并记录缺少的授权或回滚条件。

### 4. 执行演练

像未来代理一样执行目标技能：

1. 遵循目标技能的开始检查。
2. 使用它建议的工具和伴随技能。
3. 保留原始证据和精确状态标签。
4. 技能要求调用另一个技能时，验证交接输入和输出是否匹配。
5. 步骤无法继续时，先尝试技能写明的下一条恢复路径，再判断阻塞。
6. 记录技能哪里有帮助、哪里含糊、哪里重复、哪里不安全、哪里缺步骤、哪里无法恢复。
7. 区分“接口或组件能跑”和“整个业务流程完成”。如果演练只证明端点、组件或源码路径，标为部分，并说明未测外层流程。

如果演练发现缺口，修补目标技能并重跑最小受影响部分。重复直到技能完成任务，或到达真实且已记录的边界。

### 5. 验证并记录

运行技能类型对应的校验，例如 Codex 技能使用 `quick_validate.py`。

最终报告必须包含：

- 目标技能。
- 演练请求。
- 触碰的真实系统。
- 收集到的证据。
- 修改的文件。
- 生成的产物。
- 遇到的安全边界。
- 验证命令和结果。
- 仍未测试的项目及原因。

不要为了记录而创建无必要的持久审计文件。只有项目已有约定、用户要求或产物本身需要留档时，才新增审计记录。不要删除证据、日志、下载或进程，除非有直接证据证明它们属于当前演练且清理安全。

## 输出格式

报告结果时使用：

```markdown
## 技能改进结果

- 目标技能：
- 演练内容：
- 闭环结果：是/否/部分
- 证据：
- 技能变更：
- 剩余缺口：
- 验证：
- 清理：
```

当技能链到达真实安全或授权边界，但已证明边界前路径时，使用 `部分`。

## 反模式

- 不要因为 `SKILL.md` 能解析就称技能完成。
- 不要只靠当前对话总结当测试证据。
- 不要只用静态文本审查测试浏览器、数据库或 IDE 类技能。
- 不要强迫每条技能链穿过所有可能层面；真实任务没有 SQL、DML、浏览器面或旧系统运行时，应记录为 `不适用` 或真实环境边界。
- 不要把一次性本地业务请求写进通用技能；应抽象成可复用规则。
- 不要复制另一个技能的详细流程；增加交接契约。
- 不要抹掉强状态标签，例如 `needs-data`、`runtime-stale`、`manual-auth-required` 或 `blocked-needs-safe-sample-or-rollback-plan`。
- 不要清理文件、日志、下载或进程，除非直接证据证明它属于当前演练且清理安全。
- 不要默认使用子代理做验证；只有用户授权、任务适合并行或验证收益明显高于噪声时才使用。使用时必须传原始任务和产物，不泄露预期答案。
